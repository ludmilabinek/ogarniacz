# Lessons Learned

> Append-only register of recurring rules and patterns. Re-read at start by /10x-frame, /10x-research, /10x-plan, /10x-plan-review, /10x-implement, /10x-impl-review.

## Parametrize tool selection via `cmd_template`; never hardcode literals

- **Context**: starter cards in `.claude/skills/10x-tech-stack-selector/references/starter-registry.yaml` — when adding, editing, or handing off to `/10x-bootstrapper`.
- **Problem**: The Spring card had `type=maven-project` as a literal in `cmd_template`. A hand-off with `package_manager: gradle` was silently ignored — the bootstrapper only substitutes `{pm}` and `{name}`; any other literal bypasses the override mechanism. Result without a manual patch: a Maven scaffold instead of Gradle, even though the selector recorded the opposite.
- **Rule**: If a starter CLI has a flag selecting build-tool / package-manager / project variant — **always** parametrize it via `{pm}` in `cmd_template`. Never inline `maven`, `npm`, `pip`, `bundle`, `poetry`, etc. as literals. When adding a new card, audit every command argument — if it's an ecosystem choice, it must go through a placeholder.
- **Applies to**: all (mainly plan-review and implement when editing the starter registry)

## PRD-mandated capabilities must land in `tech-stack.md` as explicit fields, not just prose

- **Context**: `/10x-tech-stack-selector` when producing `context/foundation/tech-stack.md` from a PRD — especially for capabilities the PRD names as "must-have" (database, auth provider, queue, cache, external AI/LLM, storage).
- **Problem**: The PRD explicitly required "server-side database persistence", yet the resulting `tech-stack.md` never named the engine — it only mentioned "Spring Data" and "relational persistence" in the prose of `## Why this stack`. The frontmatter had no `database` field. If the gap hadn't been caught before `/10x-bootstrapper`, the scaffold would have shipped with the default H2 in-memory DB and forced a migration after the fact. The same blind-spot pattern hits every capability the PRD mandates but the starter card picks generically.
- **Rule**: For every capability required by the PRD (database, auth provider, queue, cache, external AI, file storage, search), the selector must record a **concrete engine/provider name** in the frontmatter as its own field (`database:`, `auth_provider:`, `queue:`, `ai_provider:`, etc.) and justify the choice in prose. "Spring Data" / "relational persistence" is a category, not a choice. If the PRD says "must have X", `tech-stack.md` must answer "X = specifically Y, because Z" — in a frontmatter field, not only in narrative.
- **Applies to**: all (frame and plan-review catch missing fields; impl-review verifies PRD ↔ tech-stack alignment before implementation starts)

## Pre-flight: verify that hardcoded `dependencies` in `cmd_template` cover PRD requirements

- **Context**: the pre-flight step right before `/10x-bootstrapper`, for every starter card with a hardcoded dependency / module list in `cmd_template` (e.g. Spring Initializr `-d dependencies=`, similar patterns in other frameworks with "pick starter + features" generators).
- **Problem**: The Spring card had `-d dependencies=web,devtools` hardcoded. The PRD required DB persistence, auth, and validation — none of those dependencies were in the command. A `tech-stack.md` with `database: postgresql` has no channel into `-d dependencies=...`; the bootstrapper only substitutes `{pm}` and `{name}`. Without a pre-flight edit, the scaffold ships a compiling project that **contains none of the PRD-required capabilities** — and this is misleading because "build succeeded" gives a false sense of a finished scaffold. It's a variant of the same disease as the `type=maven-project` lesson, but on a different parameter: a literal in `cmd_template` bypasses every hand-off override.
- **Rule**: Before `/10x-bootstrapper`, walk through the chosen card's `cmd_template` and compare **every hardcoded dependency / module / feature list** with the capabilities derived from the PRD and the fields in `tech-stack.md` (database, auth, validation, security, AI provider, queue, cache, storage). Add missing entries to a project-local copy of the card (`.claude/skills/.../starter-registry.yaml`) **before** scaffolding and commit the card diff together with the scaffold — that's the audit trail for why the card diverges from upstream. Don't rely on the bootstrapper "figuring it out" from the hand-off — the contract is narrower than it intuitively appears.
- **Applies to**: all (plan and implement mostly; impl-review should check that the card used during bootstrap matched tech-stack and PRD)

## The `{name}` placeholder in `cmd_template` has a dual meaning — don't use it for project-identifier parameters

- **Context**: starter cards for tools where the scaffold directory name is **independent** of the project identifier inside the generated files — Spring Initializr (`artifactId`), Maven archetypes (`-DartifactId`), Yeoman generators with a separate `--name`, any generator returning a tarball/zip without a top-level dir.
- **Problem**: The bootstrapper substitutes `{name}` as a *directory marker* depending on `cwd_strategy` — `.bootstrap-scaffold` for `subdir-then-move`, `.` for `native-cwd`. In the Spring `cmd_template`, `{name}` also sat inside `-d artifactId={name}`. Both substitutions produce illegal Maven artifactIds (`.bootstrap-scaffold` and `.` are both rejected by Initializr). The card looks correct on a quick read — `{name}` is a "documented placeholder" — but it's structurally hostile to the bootstrapper's contract. The same problem will hit any card that tries to wire `{name}` into a field with a syntactic rule stricter than "directory name".
- **Rule**: In `cmd_template`, use `{name}` **only** as the CLI argument for the target directory (e.g. `create-vite {name}`, `rails new {name}`, `django-admin startproject {name}`). For parameters identifying the project inside the generated files (artifactId, package name, app name — anything with a validation rule stricter than "directory name"), inline a **literal** (e.g. `artifactId=app`, `--package=core`) — unless the registry has a dedicated, semantically distinct placeholder (`{project}`, `{artifactId}`). When code-reviewing a new card, read `cmd_template` in both modes simultaneously: "what will tar / curl / cd see" and "what will end up in the generated project files" — if `{name}` lives in both contexts, one of them will break.
- **Applies to**: all (mostly when adding/editing cards in `starter-registry.yaml`; plan-review and impl-review on any change to the placeholder contract)

## Don't pin an ephemeral ecosystem version in a shared `cmd_template` — upstream defaults sensibly and rejects stale values

- **Context**: fields in `cmd_template` that take a framework / runtime version (Spring `bootVersion`, Node `--node-version`, `dotnet new --framework`, `cargo new --edition`, framework majors). In shared starter cards in `starter-registry.yaml`.
- **Problem**: The Spring card had `bootVersion=3.4.0` as "defensive pinning". Spring Initializr enforces an active compatibility floor — it responded HTTP 400 with `Invalid Spring Boot version '3.4.0', Spring Boot compatibility range is >=3.5.0`. `tar` received an HTML error instead of a tarball → `Unrecognized archive format`, scaffold crashed. The framework-version pin aged out faster than the time between committing the card and its first use. The assumption "pinning = stability" is false when upstream actively enforces recency through a compatibility window shorter than the registry's maintenance cadence.
- **Rule**: In a shared `cmd_template`, **do not pin** the framework / runtime version when upstream has a sensible default GA target and actively rejects stale values — Spring Initializr, `create-next-app`, `cargo new`, `dotnet new`, most modern scaffolders. Omit the parameter, trust upstream. Pin **only** when: (a) the project genuinely must sit on a specific LTS / pinned major and (b) that constraint belongs in `tech-stack.md` (hand-off, project-specific), not in the shared card. "Stable years out" versions like Java LTS (`javaVersion=21`) are fine — the discriminator is: support window measured in years, not months. Before pinning, ask: "what happens to this card in 6 months without a touch?" — if the answer is "it'll fail with HTTP 400", don't pin.
- **Applies to**: all (mostly when adding/editing cards in `starter-registry.yaml`; plan-review on code-review of new cards — catching ephemeral pins is a classic taste check)

## Verify `./gradlew test` post-bootstrap; if `DataSourceBeanCreationException` → add H2 test-runtime before first commit

- **Context**: `10x-bootstrapper` Spring Boot card — between scaffold generation and the first commit.
- **Problem**: Scaffold tests are broken-from-day-one and nobody notices. `10x-bootstrapper` does not run `./gradlew test` after scaffold; `AppApplicationTests.contextLoads()` fails immediately with `DataSourceBeanCreationException` because `spring-boot-starter-data-jpa` + `postgresql` runtime driver require a DataSource at context-load time and the test classpath has no embedded fallback. The broken test only surfaces days later when someone tries to wire CI (`./gradlew test` in GitHub Actions), and at that point the failure looks like a CI issue rather than a scaffold defect.
- **Rule**: After scaffolding a Spring Boot project, always run `./gradlew test` before the first commit. If `AppApplicationTests` fails with `DataSourceBeanCreationException` (data-jpa + non-embedded JDBC driver on the classpath), add `testRuntimeOnly 'com.h2database:h2'` to `build.gradle` so `@SpringBootTest` can boot a context without production credentials. Verify with a second `./gradlew test` returning `BUILD SUCCESSFUL` before committing.
- **Applies to**: implement, impl-review

## Per-controller `@SpringBootTest` class is the test layout standard

- **Context**: adding tests for any new `@Controller` / `@RestController`.
- **Problem**: S-01 folded controller tests into `AppApplicationTests`. S-02 split them into `EventControllerTest`. Without a rule, S-03+ will keep zig-zagging.
- **Rule**:
  - Each controller gets its own `<Name>ControllerTest` (`@SpringBootTest` + `MockMvc`, or `@WebMvcTest` if no full context is needed).
  - `AppApplicationTests` stays minimal — `contextLoads()` smoke test only. Do not add controller assertions there.
  - Spring's context cache reuses the same context across classes with identical config — splitting is ~free. The cache fragments when you introduce `@MockBean`, `@TestPropertySource`, or different active profiles; only diverge config when the test actually needs it.
- **Applies to**: `/10x-plan`, `/10x-implement`, `/10x-impl-review` on any change that adds a controller or controller tests.

## When lifting a test helper, sweep sibling `@BeforeEach` / setup blocks for duplication too

- **Context**: extracting a shared test-side helper into `LlmTestFixtures` (or any test-only fixture class) — and similar refactors across `*Test` classes that share infrastructure.
- **Problem**: Phase 1 of `llm-extraction-regression-harness` lifted `chatResponseOf` out of `LlmVisionClientTest` into `LlmTestFixtures` so the new `LlmExtractionRecordedRegressionTest` could reuse it. But the same two classes also carried an identical 4-line `@BeforeEach void stubDefaultOptions()` (`when(chatModel.getDefaultOptions()).thenReturn(ChatOptions.builder().build())`) — exactly the same shape, exactly the same Mockito stub, exactly the same dependency need. It was not lifted, because the lift was triggered by reuse-of-one-call, not by an audit of the wider surface. The duplication only surfaced in impl-review (F3), one step too late for the cheap fix.
- **Rule**: When you lift the first shared helper out of a test class, scan **all** of that class's `@BeforeEach` / `@BeforeAll` / `private` helper methods against the new sibling test that just appeared. If anything else is byte-for-byte duplicated (or a one-symbol rename away), lift it in the same change. A second `*Test` file is a strong signal — the pair forms a *family*; treat the family's fixture surface as a single thing. Without this discipline, every additional sibling adds another N-way duplication round. The scan covers the **entire** shared fixture surface, not just `@BeforeEach`: identically configured `@MockitoBean` / `@MockBean`, shared `@TestPropertySource` properties, `@TempDir` setup, and custom AssertJ matchers all count as the same category of duplication and lift the same way.
- **Applies to**: `/10x-plan` (when a plan introduces a sibling test class alongside an existing one), `/10x-implement`, `/10x-impl-review` (verify the fixture-class scan happened).

## Fly.io creates an HA peer on the first `fly launch`/`deploy` despite `min_machines_running = 1`

- **Context**: deploy-plan execution on Fly.io, the phase right after the first `fly deploy` (or `fly launch`) — stack-agnostic. Applies to any app whose `fly.toml` declares `min_machines_running = 1` and the operator assumes that literally means "one machine".
- **Problem**: Since ~2024, Fly defaults to creating a second Machine as an HA peer on the first launch/deploy, **ignoring** `min_machines_running = 1` from `fly.toml`. That field only affects auto-stop (when `auto_stop_machines = true`), not initial provisioning. `flyctl status` then shows 2 Machines `started` in the same region. Consequences: 2× compute bill, 2× HikariCP pool to Neon (a non-issue at the free tier's 100 connections, but it can bite after upgrading to Launch), and an undocumented inconsistency between `fly.toml` ("intent: 1 machine") and runtime ("actual: 2 machines"). Easy to miss because `fly.toml` looks fine and `fly deploy` ends in success.
- **Rule**: After the first `fly deploy`, always check `flyctl status -a <app>`. If you see more machines than you want (typically 2 instead of 1 for an MVP), run `flyctl scale count 1 -a <app>` — that's the authoritative way to set machine count. Then add `[deploy] strategy = "immediate"` to `fly.toml` along with a comment documenting the intent, so a future `fly deploy` or re-launch won't quietly recreate the peer and the next operator sees plainly that deploy-time downtime is an accepted trade-off. The deploy-plan verification step must explicitly distinguish "1 Machine started" (expected) from "2 Machines started" (HA peer to remove) — not just "started" as success.
- **Applies to**: deploy, infra-research, plan-review (any change to the deploy section or `fly.toml`)
